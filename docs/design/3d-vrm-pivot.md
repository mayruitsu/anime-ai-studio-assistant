# 3Dモデル（VRM）ポージング方針への転換メモ

> このドキュメントは、フェーズ6・7（実写真・イラストの2D剛体パーツ切り抜き）で見えた限界を受けて、プロジェクトの技術方針を大きく転換した経緯と、新方針の内容を整理するものです。

## 経緯：2D剛体パーツ切り抜きで何が起きたか

フェーズ6のMVP（[docs/design/character-animation-concept.md](character-animation-concept.md)）で、1枚の画像から関節を検出し、パーツごとに切り抜いた画像を回転・合成する仕組みを実装した。軽い動き（肘を軽く曲げる程度）では動作したが、実際にダイナミックな動き（ジャンプ、体をまたぐポーズ、両脚を大きく開く等）を試したところ、以下の問題が解決できないことが分かった。

- **腕がちぎれて見える**：関節間の距離が元写真と変わると、パーツ同士の継ぎ目に隙間ができる（伸縮・キャンバス拡張で軽減はしたが根治はしない）
- **肩・肘・手首が密集するポーズで破綻する**：関節が近すぎるとパーツの境界判定が不安定になる
- **両脚をまたぐ衣服（スカート等）が割れて見える**：脚を独立に回転させる設計上、避けられない

これらは実装の細かい不具合ではなく、**「画像を矩形パーツに切り分けて剛体のまま回転させる」という手法そのものの限界**であると判断した。この手法は昔ながらの切り紙アニメと同じ原理であり、業界的にもLive2DやSpineが「メッシュ変形・ウェイトペイント」という技術を発明してこの限界を乗り越えてきた、という経緯がある。

## 検討した代替案

| 案 | 却下 / 採用 | 理由 |
|---|---|---|
| Live2D的な2Dメッシュ変形を自作 | 保留 | 実現可能だが、既製3Dモデルを使う案の方が早く「動くもの」に到達できると判断 |
| 写真から3D復元AIを自作（SMPL相当） | 却下 | 数千人規模の3Dスキャンデータセット・統計的形状モデリング・大規模学習が必要で、個人開発の規模を大きく超える |
| 既製の3D復元AI（PyMAF、PIFuHD等）を利用 | 保留（トラックBで検討） | 実写真専用（イラスト非対応）、体型推定が粗い、服の再現度が低い、見えない部分のテクスチャがそもそも存在しない、導入負荷が高いなど課題が多い |
| **既製の3Dモデル（VRM）を使い、ポージング・アニメーション機能を先に作る** | **採用** | 3Dモデル生成という一番難しい部分を後回しにでき、確立された技術（3D骨格アニメーション）だけで「動くもの」を早く作れる |

## 新方針：2トラック体制

### トラックA：3Dモデルポージング＆アニメーションアプリ（優先）

「精巧な3Dモデルがすでにある」という前提を置き、既製のアニメ調3Dアバター（VRM形式）を読み込んで、関節をポインターで操作し、アニメ的なレンダリング・タイミングで動画化するアプリを作る。

- **VRM形式**：VRoid Studio等で作られたアニメ調3Dアバターの標準フォーマット。VRoid Hub等で無料モデルが多数配布されている
- **技術スタック**：フロントエンドをThree.js＋`@pixiv/three-vrm`で構成し、VRMの読み込み・骨格操作を行う（既存のReact構成に3D描画を追加する形）
- **アニメらしさの出し方**：
  - レンダリングはセルシェーディング（トゥーンシェーディング）で平面的な陰影にする
  - フレームをなめらかに全部出力せず、間引いて出力する（2コマ・3コマ打ち）ことで、3Dモデルの滑らかさを抑えてアニメ的なカクつきを出す
- 「本物の写真ピクセルを使う」という当初方針からは外れるが、「生成AIで新しい画像を作らない」という方針（3Dモデルの操作・レンダリングであり、画像生成ではない）は維持できる

### トラックB：写真から3Dモデルを作る研究（並行・非ブロッキング）

トラックAの完成を待たず、並行して「ユーザー自身の写真から3Dモデルを作る」方法を探る。既製の3D復元AIモデル（PyMAF、PIFuHD等）を試し、実写真でどの程度の精度が出るか検証するところから始める。トラックAの進行をブロックしない、時間のかかる探索的タスクとして扱う。

## 既存の2Dパイプライン（フェーズ6・7）の扱い

`segmentation.py` / `part_cropping.py` / `rotation.py` / `render.py` / `multi_photo.py` 等、フェーズ6・7で作った2D剛体パーツ切り抜きの実装は、技術検証として意味があったため**削除はしない**。ただし今後の開発の主軸はトラックA・Bに移るため、ロードマップ上は「一旦保留」として扱う。

## ドキュメント・ルールへの影響

- `CLAUDE.md`のプロジェクト概要・技術スタックは、トラックAの内容を反映して更新する
- `docs/roadmap.md`にトラックA・Bを新しいフェーズとして反映し、フェーズ6・7は保留ステータスに変更する
