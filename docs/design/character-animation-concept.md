# キャラクターアニメーション構想メモ（検討中）

> このドキュメントは考えを整理するための作業メモです。結論が出たら `docs/roadmap.md` に正式なタスクとして反映します。

## 目的

「1枚の画像から自動でポーズ推定→点をドラッグ」という現状の仕組みを発展させ、**自然な人物アニメーションを、画像で学習した生成AIを使わずに作れるようにする**。

## 基本方針（AIの使い方の整理）

これまでの議論で、AIの使いどころを以下のように切り分けた。

| やりたいこと | 使うもの | 画像学習した生成AIか |
|---|---|---|
| 元画像から関節点を検出する | MediaPipe Pose（既存） | ×（座標を返すだけで画像は生成しない） |
| 人物を背景から切り抜く | セグメンテーションモデル | ×（同上） |
| ポーズを変えたときの見た目を作る | **写真パーツの切り貼り・変形**（本メモの主題） | ×（本物の写真ピクセルを使う） |
| フレーム数の管理・背景の切り替え | UI操作 | 不要 |
| 背景のパン/ズーム、回転演出 | 幾何変換（アフィン変換・樽型歪み） | 不要（後回し） |

→ **生成AI（TPSMM/FOMMのような学習済みモデルで不足部分を"想像"で埋める手法）は使わない方針に転換。** 理由：複数枚の実写真があれば、想像で埋める必要がそもそもない。

## コンセプト：切り絵（紙人形）アニメ × 3Dモデリング

> 当初「Live2Dのような」という表現をしていたが、Live2D特有の「パーツがラバーのようになめらかに曲がる」見た目は**アニメらしさと合わないため不採用**。パーツは内部変形させず**剛体のまま回転・平行移動のみ**とする。関節でカクッと切り替わる、紙人形・切り絵アニメ的な見た目を狙う。

- **切り絵アニメ的な要素**：パーツ（頭・上腕・前腕…）は内部で変形せず、剛体として関節を軸に回転・配置するだけ。関節の継ぎ目のカクつきはアニメらしさとして許容する
- **3Dモデリング的な要素**：素材を「絵」ではなく複数枚の実写真から作る。パーツ同士の重なり順・可動域を、単純な2Dリグより一段階リッチな構造（3D的な位置関係の把握）で管理する

Live2Dとの違いは、①パーツ素材が「実写真から切り出したもの」である点、②パーツ内部を変形させず剛体のまま動かす点。3Dモデリングとの違いは、最終出力があくまで2D画像であり、フルの3Dメッシュ・テクスチャを持つわけではない点。

### 使い分けのイメージ（要すり合わせ）

- **モデル生成フェーズ**：4枚程度の写真（正面・腕を上げた状態、など異なるポーズ/角度）から、パーツ画像＋関節構造を作る。ここが「写真から3Dモデル的なものを作る」に近い作業
- **ポージングフェーズ**：作った関節構造を動かして姿勢を決める。パーツは剛体のまま回転・配置するだけ
- **レンダリングフェーズ**：決めた姿勢に応じて、パーツ画像を配置・合成して1フレームの画像を出力する（内部変形はしない）

## 表情・髪など「剛体回転」だけでは足りない部分

体パーツ（頭・上腕・前腕…）は剛体回転で対応できるが、以下は性質が違うため別扱いが必要。

- **表情変化**：関節ではなく「状態のバリエーション」の問題。体パーツと同じ**写真の状態違いを複数撮って差し替え/合成**する方式で対応（口の形・目の開閉など）。AI不要、伝統的なアニメ手法そのもの
- **髪の動き**：関節を持たず風や慣性で連続的に揺れるため、写真の差し替えでは対応しきれない。→ **古典的な物理シミュレーション**（バネ・振り子モデル）で髪の房を揺らす。AI不要、単なる数値計算。Live2Dの髪・服の揺れも同じ仕組み（AIではなく物理演算）

→ 「学習済みAIに人体の動き・重力・美術知識を学習させて、写真にない姿勢を予測変形する」という案も技術的には可能だが、これは結局「画像/動画学習した生成AI」に戻るため、今回は見送り、上記の物理シム＋写真差し替えの組み合わせで一貫性を保つ。

### 物理シミュレーションとアニメ的フレーム間引きの関係

物理演算は内部的には細かい時間刻みで計算する必要があるが、出力（レンダリング）は間引いてよい。つまり：

- 髪の物理シムは内部で滑らかに計算する
- 実際に絵として出すフレームは、2コマ・3コマ打ちのように間引く

これにより、**体のポーズ（剛体回転）と髪の物理シムを、同じ「出力フレームレート」設定で一括して間引ける**。下記の論点6（フレーム補間方式）は、体だけでなく髪の物理シムにも共通して適用される設定になる。

## 未確定事項・要検討ポイント

1. **「3Dモデリング的」の具体度**：本当に簡易3Dメッシュ（深度情報あり）を持つのか、それとも2Dパーツに前後関係（重なり順）だけを持たせた疑似3D（2.5D）で十分か
2. ~~パーツ分割の粒度~~ → **決定：パーツは剛体のまま回転・配置。内部変形（メッシュワープ）はしない。**Live2D的ななめらかな変形は不採用
3. **複数写真の使い分けロジック**：目標ポーズに一番近い写真のパーツを選ぶのか、近い2枚をブレンドするのか
4. **写真間のズレの扱い**：光の当たり方や体の向きが写真ごとに違う場合、パーツ境界の継ぎ目をどう自然に見せるか
5. **可動域を外れた場合の挙動**：4枚の写真でカバーしていない極端なポーズを指定されたときにどうするか（作らない/近似で我慢する/警告する）
6. **フレーム補間の方式**：フレーム間をなめらかに補間する（滑らか）か、アニメの2コマ・3コマ打ちのようにあえてカクつかせるか。②の剛体回転・髪の物理シムと合わせて「アニメらしさ」を左右する軸

## 優先順位：最低限の実装手順（MVP）

**目標：1枚の画像が変形して、たとえつたなくても動画としてダウンロードできる状態を最短で作る。**

前提：1枚の写真のみを使う。複数写真での使い分け・表情差し替え・髪の物理シム・背景演出は今回のスコープ外（後述の「後回しリスト」に整理）。

1. **人物切り抜き（セグメンテーション）** — 新規。MediaPipe Selfie Segmentationなどで人物マスクを取得（既存の`pose.py`と同じ構成で`segmentation.py`的なものを追加）。マスクで人物だけ抜いておくことで、パーツを動かした後に元の位置へ"残像"が残る問題を避けられる
2. **パーツ分割の定義** — 軽い新規実装。既存の33点（POSE_CONNECTIONS）を使い、体を大まかなセグメント（頭胴体・上腕×2・前腕×2・太もも×2・すね×2くらい）にグルーピングする対応表を作る。検出処理自体の追加は不要
3. **パーツ画像の切り出し** — 新規。①のマスクと②の対応表を使い、各パーツを回転ピボット（付け根の関節）付きの画像チップとして切り出す
4. **ポーズ→回転角の計算** — 新規。既存のドラッグ操作（`PoseCanvas.jsx`）で得られる点の新しい位置から、各パーツの回転角度を算出するロジック
5. **フレーム合成（レンダリング）** — 新規。フレームごとに、③のパーツ画像を④の角度で回転させ、透明/単色背景の上に組み立てて1枚の合成画像を作る
6. **動画出力の差し替え** — 既存`video.py`の改修のみ。「元画像＋赤い点」を描画している部分を、⑤で作った合成済みフレーム画像に差し替える

キャラクター単体が単色/透明背景の上で動く映像がダウンロードできれば、このMVPの達成とする。実写真背景との合成は含めない。

## 後回しリスト（MVP達成後にまとめて着手）

MVPの6ステップが完了してから、優先度を検討し直して着手する。

- 複数写真でのモデル化・パーツの使い分け（目標ポーズに近い写真を選択/ブレンド）
- 写真間のズレ（光の当たり方・体の向き）の吸収
- 表情変化（口・目の状態差し替え）
- 髪の動き（物理シミュレーション）
- フレーム補間方式の作り込み（2コマ・3コマ打ちなどアニメ的な間引き）
- 背景との合成、フレームごとの背景設定
- 背景のパン・ズーム・回転演出（魚眼歪み）
- 背景の一部変化・レイヤー分けパララックス
- 可動域を外れたポーズを指定された場合の挙動
- （将来）ここまでの仕組みをベースにした自作モデル化の検討

いずれも技術的な方向性は本メモで検討済み（幾何変換・物理シム・写真差し替えで対応、AI不要）。実装順序はMVP完了後に改めて決める。
